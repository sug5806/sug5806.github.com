2019-04-07 TCP/IP 4계층

2019/04/07 카테고리 네트워크

OSI 7 Layer 와 TCP/IP

  • 응용 계층(Application Layer) = Application
  • 표현 계층(Presentation Layer)
  • 세션 계층(Session Layer)
  • 전송 계층(Transport Layer) = Transport
  • 네트워크 계층(Network Layer) = Internet
  • 데이터 링크 계층(Data link Layer) = Network Interface
  • 물리 계층(Physical Layer)

TCP/IP에서 알아야할 프로토콜

응용 계층(Application Layer) = FTP, HTTP, SSH
전송 계층(Transport Layer) = TCP, UDP
네트워크 계층(Network Layer) = IP
데이터 링크 계층(Data link Layer) = Ethernet

TCP/IP 4계층

네트워크 전송 시 데이터 표준을 정리한것이 OSI 7 Layer이고 이 것을 바탕으로 실제 사용하는 인터넷 표준이 TCP/IP 4계층이다.

4 계층 : Application (OSI 7Layer의 5, 6, 7계층 담당)

  • 주요 프로토콜
    • FTP
    • Telnet : 보안에 너무 취약하여 현재는 안쓰임
    • DNS(Domain name System) : IP와 Name(Domain)을 이어주는 역할
      • 사용자가 보내면 request를 받아야하는데 이 역할을 하는게 DNS server
    • HTTP
      • http://www.naver.com///~/
      • schema/host/ /디렉토리들/
      • 1.1 부터는 clnt랑 server랑 한번 연결하면 필요없을때까지 연결을 끊지 않음

3계층 : Transport (OSI 7Layer의 4계층 담당)

port(socket에 부여된 번호, port는 프로세스안에 있음)번호를 관리함

  1. TCP(E-mail, 채팅)
    • 신뢰도 높음
      • timeout이 났을 경우 자동으로 재전송
    • 연결 지향적
      • 상대방에게 패킷을 보내고 받았는지 체크(ACK)
    • TCB
      • 서로 연결이 되면 TCB에 등록을 해놓음
      • 소켓쌍 - (source IP, source port), (dest IP, dest port)
      • send buffer, receive buffer에 대한 포인터
        • send와 receive -> 커널 영역
    • sliding window
      • receive window : 내가 받을 수 있는 패킷의 양
    • 3 Way handShaking
      • byte stream에 첫번째 부여하는 숫자를 ISN(initial sequence number)이라 한다
      • 그 숫자를 sequence라고 함
        1. client가 server에 SYC와 자기가 보내고자한 바이트의 seq를 같이 보냄
        1. server역시 SYC + ACK(상대방이 보낸 seq에 +1을 해서)를 보내 자신의 send buffer와 client의 receive buffer와 동기화 시킴
    • TearDown
        1. client가 FIN를 보냄
        1. Server는 ACK를 보냄 -> clinet는 send는 못하지만 receive는 가능, server가 남은 데이터를 send함
        1. 남은 데이터를 보내고 난뒤 server가 FIN를 보냄 (Time Out만큼 기다림)
        1. client가 server의 FIN을 받았으므로 ACK을
  2. UDP(스트리밍 서비스, 게임)
    • 신뢰도 낮음
      • 상대방에게 패킷을 보내도 받았는지 체크 안함
    • 비 연결 지향형 프로토콜
    • 데이터를 재전송하지 않는다.
    • 전송된 데이터 일부가 손실될 수 있다.

2계층 : Internet(OSI 7Layer의 3계층 담당) -> IP routing을 함, IP address를 다루는 프로토콜

  • IP routing : source에서 desti로 갈때 어느 router로 갈지 결정하는 것
    • 라우팅 테이블을 통하여 어느 route로 갈지 등록을 해놓는다
    • netstat -r을 했을때 나오는 Destination은 네트워크 IP이다
    • default는 라우터의 IP
    • 한번 ARP로 하면 같은 LAN 상에서는 서로의 MAC주소를 알고 있으니 게이트웨이가 MAC으로 잡힌다
  • ISP(Internet Service Provider) : KT, SK같은 인터넷 공급자들
  • IP
    • 하나의 IP주소에는 Network ID와 Host ID가 존재하고있음
    • Source Address : 4 bytes, 송신 IP
    • Destination Address : 4 bytes, 수신 IP
    • TTL(Time To Leave) : router를 하나씩 거칠때마다 줄어들어 0이 되면 패킷은 버려진다
    • IP는 network 주소와 host주소로 구분된다
      • network주소 : router를 찾는 주소(LAN)
      • host주소 : 찾은 LAN에 소속되어있는 특정 host를 찾는 주소(=컴퓨터)
    • Subneting
    • public IP(공인 IP) -> IP를 돈을 주고 삼(SK, KT 등)
    • private IP
      • Ex) 커피숍에 공인IP하나를 할당해주고 router가 허용하지 않은 대역대 내에서 마음대로 나머지 고객들에게 private IP를 할당함
      • public IP와 private IP를 통신하게 해주는 것이 NAT, NAPT
        • NAT : 사설 IP로 연결되어있는 IP가 외부로 갈경우 매핑되어있는 테이블과 IP를 바꿔 보내고 다시 받으면 테이블의 IP로 다시바꾸고 그것을 다시 ARP로 바꿔 송신받는 기술
        • NAPT(Network Address Port Translation) : NAT에 추가하여 port번호까지 전송
  • BroadCast : LAN상에 있는 모든 사람들에게 패킷을 보내 MAC주소를 알아냄
    • ARP(Address Resolution Protocol)(프로토콜) -> ip를 MAC주소로 변환
      • 같은 LAN 상에있는 호스트들에게 ping요청을 보내 호스트들의 ip와 MAC주소를 매핑시키는 작업
      • Network Interface와 Internet 사이에 있음

1계층 : Network Interface(OSI 7Layer의 1,2계층 담당) : MAC주소

  • Tx에서 쏜 데이터를 Rx에서 받아야함(Serial 통신) -> Data Link 계층
  • NIC(network interface card) : 일반적으로 랜카드라 불림. 네트워크 어댑터
  • MAC(Media Access Control) : NIC의 하드웨어 주소, 앞의 3자리는 제조사 뒤 3자리는 NIC 번호
  • Ethernet(프로토콜) -> lan에서 host를 구별하기위함
    • Destination MAC Address : 6bytes, 패킷을 수신 하는 NIC -> LAN에서의 router
    • Source Mac Address : 6bytes, 패킷을 송신하는 NIC
    • MTU(한번에 보낼수 있는 데이터의 양) -> 1500bytes : 헤더를 제외하면 1350정도

LAN(Local Area Network) : 보통 router를 기준으로 묶여있는 것(같은 router안에서 host를 구분하기위해 MAC주소를 사용함), 같은 LAN상에서 데이터를 보낼때도 이용
WAN(wide Area Network) : IP를 사용

하드웨어 주소 = 물리주소 = MAC주소 = NIC주소 = LAN카드 주소 = 제2계층주소 = 데이터링크계층 주소
이더넷 주소는 하드웨어주소의 일종입니다.

논리주소 = 네트워크 주소 = 제3계층주소 = 프로토콜 주소
IP주소는 논리주소의 한 종류입니다.

2019-04-07 Synchronous와 Asynchronous의 차이점

2019/04/07 카테고리 운영체제

Synchronous(동기)와 Asynchronous(비동기)의 차이점

동기식과 비동기식의 큰 차이점은 데이터를 받는 방식에 있습니다

동기식 처리는 A,B,C 순서대로 작업이 수행되었을 경우 A,B,C의 순서로 작업이 끝나야 합니다. 작업의 순서가 보장됩니다.

Synchronous란 '동시에 일어난다’라는 뜻을 가지고 있으며 여기서 동시에 일어난다는 것은 요청에 따른 응답이 동시에 일어난다는 것입니다.


출처 : https://nesoy.github.io/articles/2017-01/Synchronized

반면 비동기식 처리는 작업의 순서가 보장되지 않습니다. A,B,C의 순서로 작업이 시작되었더라도 A,B,C의 순서로 작업이 끝난다고 보장 할 수 없습니다.


출처 : https://nesoy.github.io/articles/2017-01/Synchronized

2019-04-05 HTML 기초태그

HTML 기초 태그

  • !DCTYPE html : 이 문서를 HTML5로 정의하겠다
  • html : html의 루트 요소
  • head : 문서에 대한 메타정보를 포함
  • title : 이 문서의 타이틀을 명시
  • body : 이 문서의 시각적인 페이지 컨텐츠 포함
  • h1 : heading을 정의
  • p : 단락

HTML의 태그는 <tagname>content</tagname> 으로 이루어져 있다

HTML Documents

  • 모든 HTML문서는 !DOCTYPE html로 타입정의를 하고 시작한다
  • HTML 문서 자체는 html로 시작하고 /html로 끝난다
  • HTML의 비주얼적인 부분은 body와 /body 사이에 작성한다

HTML Headings

  • HTML의 헤딩은 h1 ~ h6로 정의되어 있다
  • h1부터 h6까지는 글자 크기의 차이가 있다

HTML Links

  • HTML의 링크는 a태그로 정의되어 있다
  • href 속성에 주소를 적으면 된다.

HTML Images

  • html의 이미지는 img태그로 정의되어 있다
  • 원본파일(src), 택스트(alt), width와 height로 사진의 크기를 조절할 수 있다

HTML Buttons

  • HTML의 버튼은 button태그로 정의되어 있다

HTML Lists

  • HTML의 리스트는 ul(정렬되지 않은 리스트), ol(숫자로 정렬되는 리스트) 로 나뉘어 있다

  • IP주소의 구성 : Network 주소. Network주소. Host주소. Host주소

C 클래스 : 소규모 네트워크에 적용

  • 총 2,097,150개의 네트워크를 만들 수 있으며 각 네트워크마다 254개의 노드를 연결할 수 있다
  • IP주소의 구성 : Network 주소. Network주소. Network주소. Host주소

    • commit 메시지를 작성할때는 현재형으로 작성하는것이 좋다
    • commit 메시지를 적을때는 제목으로 무엇을 변경했는지 알아볼수 있게 작성하는게 좋다
    • collection (list, dictionary, tuple 등)을 사용할때는 나열식으로 하는것 보다는 값마다 한줄 한줄씩 하는게 좋다
    • collection을 사용할때는 마지막 원소에 ,(콤마)를 적어주면 깃에서 추가된 내용만 추가가 된다.
    • commit을 할때는 동작하는 최소 단위로 해주는 것이 좋다(function단위도 좋은 방법)

2019-03-29 컴파일러와 인터프리터의 차이점

2019/03/29 카테고리 컴파일러이론

둘 사이의 가장 단순한 차이점은 입력시점과 실행시점의 차이이다. 컴파일러 언어는 컴파일 시간과 실행시간이 따로 있어 소스코드는 컴파일 시간에 들어오고 런타임때 입력데이터가 들어온다. 그러나 인터프리터 언어는 컴파일 시간이없고 실행 시간만 있으며 소스코드와 입력데이터가 같은 시점에 들어온다는 점이다.

컴파일러

초창기의 컴퓨터는 기계어로 프로그래밍을 했다. 그러나 기계어로 프로그래밍을 하기는 너무 어려워 이를 보다 편하게 사용하기 위해 방법을 생각했다.

  1. 일정한 형식으로 작성된 문서를 기계어로 자동 번역하는 프로그램 A를 만든다.
  2. 이후 프로그램을 만들때마다 A가 번역할 수 있도록 일정한 형식으로 문서를 작성한다.
  3. 문서 작성이 완료되면 프로그램 A를 먼저 실행하고 파일을 넘겨서 A가 자동으로 번역해준 기계어 파일을 얻는다.

그리고 여기서 사용되는 프로그램 A를 컴파일러(compiler) 라 하고, 이때 작성한 일정한 형식으로 작성한 문서를 소스파일(source file) or 소스코드(source code), 그리고 이를 번역하는 행동을 컴파일(compile) 이라 한다.

그리고 시간이 흘러 파일 하나에 모든 소스코드를 작성하는 방식은 불편하다고 느껴 여러 파일에 나눠 작성을 하고 이렇게 나뉘어 컴파일된 파일을 모아 하나의 실행가능한 파일을 만들면 이것을 링크(link) 했다고 하고, 이때 사용되는 프로그램을 링커(linker) 라고 한다.
이것을 종합하면

  1. 소스 코드를 작성하고 파일로 저장한다.
  2. 저장된 파일을 컴파일러를 이용하여 컴파일한다. 목적 파일이 생성된다.
  3. 생성된 목적파일을 링크하여 링커를 통해 실행 가능한 목적 파일이 생성된다.

컴파일러와 링커 모두 목적 파일을 생성하나 컴파일러는 실행이 불가능하고 링커는 실행 가능한 목적파일을 생성한다는 점이다.

컴파일과 링크 과정을 합쳐 빌드(build) 라고 하고, 이때 사용되는 프로그램을 빌더(builder) 라고 한다.

그러나 컴파일러 언어는 단점이 있는데 기계어는 기계마다 그 내용이 다르기 때문에 똑같은 프로그램 일지라도 기계가 바뀔 때마다 해당 기계에 맞는 기계어로 다시 번역을 해야한다. 그러나 소스코드와 기계어 사이에 중간 단계의 언어가 있어 소스코드가 중간 단계의 언어로 먼저 번역된 다음, 번역된 중간 단계의 언어가 기계어로 번역된다면 위와같은 문제점을 해결할수 있지 않을까?

인터프리터

인터프리터(Interpreter) 란 컴파일 과정을 거치지 않고 소스 코드를 바로 해석하여 결과를 출력하는, 소스 코드 실행기 프로그램을 말한다. 둘 사이의 차이점을 알아보자

  1. 먼저 컴파일러는 컴파일을 하고나서 그 즉시 실행을 하지는 않는다. 그러나 인터프리터는 실행기 이므로 번역과 분석이 끝나면 바로 프로그램을 실행하여 결과를 내놓는다.
  2. 컴파일러는 기계어로 번역을 한 후 목적 파일을 생성하는 반면 인터프리터는 그렇지 않다는 점이다.
  3. 컴파일러는 컴파일하는데 시간이 걸리지만 끝나고 나면 분석이 완전히 되었으므로 다시 분석할 필요없이 바로 실행을 하면되지만 인터프리터는 실행을 할때마다 매번 분석을 진행하므로 컴파일러보다 속도가 느릴 수 밖에 없다

Procedural vs Object-Oriented

  • procedural -> 함수를 이용해서 추상화를 했다
  • Object-Oriented -> 객체를 통해 추상화를 했다
    • 객체 단위로 프로그래밍을 한다
    • 객체(Object) -> ‘관련있는’ 변수(상태정보)와 ‘관련있는’ 함수를 한곳에 모아놓은 ‘곳’ (곳 = 메모리) -> bundling
      • attribute(속성)
        • instance member(변수)
          • 각각의 객체가 가지는 값
        • instance method(함수)
          • 객체가 할 수 있는 행동, 기능

message passing

  • 다른 객체(other)의 상태정보(멤버)를 변경할 때는 반드시 상대 객체가 가진 메서드를 이용한다

클래스는 함수의 모임

객체와 인스턴스의 차이
객체와 인스턴스의 메모리는 같음
인스턴스=> Ex) me라는 객체는 Account라는 클래스의 인스턴스

OOP

  1. 캡슐화(Encapsulation)

    1. 관련있는 멤버(변수) 와 메서드(함수)를 하나의 단위(대부분 클래스)로 묶는 것
      1. 클래스를 제외하고도 하나의 단위로 묶을수 있다면 OOP라고 함
        1. Ex) C에서의 구조체
        2. the bundling of data with methods
      2. 정보 은닉을 포함함

  2. 정보은닉( 파이썬은 완벽한 정보은닉을 지원 안함) -> C++에서 클래스 만들면서 private선언

    1. 어떤 멤버를 공개 / 비공개
    2. 어떤 메서드를 공개 / 비공개

  3. 다형성

    1. 상속
      • 상속을 쓰지 말자
      • 상속은 OOP의 정수이다
        • 상속은 쓰되 다중상속은 하지말자
      • 멤버변수에는 절대 직접접근 금지
      • 객체의 멤버에 접근 -> getter / 수정 -> setter 을 수행할때는 method function을 이용해야함
      • 내부에서는 멤버처럼 쓰는것처럼 하면서 property를 사용하여 정보은닉을 함
      • 인스턴스 멤버를 만든다고하면 지금 당장 쓰지 않아도 되도록이면 생성자에서 만들어 주는게 좋음
      • 멤버는 어디에서든지 생성될 수 있음
    2. Relations between classes
      • choose either inheritance or composition (언제 상속으로하고 언제 컴포지션으로 하는가)
        • IS - A
          • A laptop IS - A computer / ~은 ~의 한 종류이다
          • Inheritance (상속)
          • 조건 1 : 모든 멤버와 모든 메서드의 중복이 되어야함
          • 재사용성(reusability) -> 재사용성 만으로 IS -A 상속을 하면 절대 안되고 조건을 모두 충족해야함!
        • HAS - A
          • A policeman has a gun / ~이 ~을 가지고 있다. or 포함하고 있다
          • Composition(합성) -> ◆(In UML)
            • same life cycle = 강하게 연결되어 있다
          • Aggregation(통합) -> ◇(In UML)
            • NOT same life cycle = weak coupling
    3. method overriding (메소드 오버라이딩) -> 상속이 필수
      • polymophism (다형성)
        • 상속 관계에 있는 다양한 클래스의 객체에서 같은 이름의 메서드를 호출할 때, 각 객체가 서로 다르게 구현된 메서드를 호출함으로써 서로 다른 행동, 기능, 결과를 가져오는 것
    4. abstract class (추상 클래스)
      • 인스턴스를 만들 수 없다.
      • 하나 이상의 abstract method (추상 메소드)
        • 몸체(body)가 없는 함수 -> pass
      • 파생 클래스에서 반드시 overriding을 해야함 -> 구현을 강제하도록 한다
