2019-04-25 데이터 베이스 기초

2019/04/25 카테고리 데이터베이스

데이터 베이스 기초

데이터 베이스 4가지 정의 : 통합된 데이터, 운영 데이터, 공용 데이터, 저장된 데이터

데이터 베이스의 특징 : 실시간 접근성, 내용에 의한 참조, 동시 공용, 계속적인 변화

데이터 베이스 시스템 : 데이터베이스를 이용하여 자료를 저장하고 관리하여 정보를 얻어내는 데 필요한 컴퓨터 중심의 시스템

분산 데이터베이스 : 물리적으로는 컴퓨터 네트워크를 통해 분산되어 있는 데이터 베이스

데이터 웨어하우스 : 급증하는 다량의 데이터를 효과적으로 분석하여 정보화하고 이를 여러 계층의 사용자들이 효율적으로 사용 할 수 있도록 한 데이터베이스

DBMS : 사용자와 데이터베이스 사이에서 사용자의 요구에 따라 정보를 생성해 주고, 데이터베이스를 관리해 주는 소프트웨어. 기존에 있던 데이터 중복성과 데이터 종속성 해결

DBMS의 필수기능 : 정의, 조작, 제어

논리적 독립성 : 논리적 구조를 변경시켜도 응용프로그램에 영향 없음

물리적 독립성 : 물리적 구조를 변경시켜도 응용프로그램에 영향 없음

데이터 사전(= 시스템 카탈로그) : 데이터베이스에 저장되어 있는 모든 데이터 개체들에 대한 정보를 유지,관리하는 시스템. 카탈로그에 저장된 정보를 메타 데이터라고 함. 좁은의미로 자료 사전이라 함.
스키마는 컴파일되어 데이터 사전에 저장됨.

인스턴스(Instance) : 속성이 구체적인 데이터 값을 가지고 있는것

외부스키마 = 서브 스키마 = 사용자 뷰 : 사용자나 응용 프로그래머가 각 개인의 입장에서 필요로 하는 데이터베이스의 논리적 구조를 정의한 것 뷰 정의를 허용하는 단계

개념 스키마 : 단순히 스키마라 하면 개념 스키마를 의미, 기관이나 조직체의 관점에서 데이터베이스를 정의

내부 스키마 : 데이터베이스의 물리적 구조, 데이터의 실제 저장방법을 기술

DDL(데이터 정의 언어) : 번역한 결과가 데이터 사전에 저장됨

DML(데이터 조작 언어) : 사용자로 하여금 데이터를 처리할 수 있게 하는 도구로서, 사용자와 DBMS간의 인터페이스를 제공함. 포트란, 코볼등의 호스트 언어에서 DB기능을 추가해서 만든 언어

DCL(데이터 제어 언어) : 데이터에 대한 액세스(권한)을 제어하기위한 데이터베이스 언어

질의어 : 단말 사용자가 쉽게 DB를 액세스 할 수 있도록 대화식의 자연어로 만든 비 절차적 조작언어

요구조건분석 : 데이터베이스를 사용할 사람들로부터 필요한 용도를 파악함

개념적 설계 : 현실 세계에 대한 인식을 추상적 개념으로 표현하는 과정 (개념 스키마 모델링과 트랜잭션 모델링을 병행수행)

논리적 설계 : 컴퓨터가 처리할 수 있는 물리적 저장장치에 저장할수 있도록 변환하기 위해 특정 DBMS가 지원하는 논리적 자료 구조로 변환시키는 과정

물리적 설계 : 논리적 설계 단계에서 논리적 구조로 표현된 데이터를 디스크 등의 물리적 저장 장치에 저장할 수 있는 물리적 구조의 데이터로 변환하는 과정.

데이터 베이스 구현 : 논리적 설계 단계와 물리적 설계 단계에서 도출된 데이터베이스 스키마를 파일로 생성하는 단계

데이터 모델 : 현실 세계의 정보들을 컴퓨터에 표현하기 위해서 단순화, 추상화하여 체계적으로 표현한 개념적 모형
개념적 데이터 모델 : 인간의 이해를 돕기위해 현실세계에 대한 인식을 추상적으로 표현하는 과정. 정보 모델이라고도 함. ex) E-R모델

논리적 데이터 모델 : 컴퓨터 세계의 환경에 맞도록 변환하는 과정. 단순히 데이터 모델이라 하면 논리적 데이터 모델을 의미

논리적 데이터 모델 종류

  1. 계층형 데이터 모델 : 트리구조를 이용하여 표현. 상위와 하위가 일 대 다의 대응관계 개체 = 세그먼트 : 트리를 구성하는 노드 정보가 전달되는 단위. 개체간의 관계를 부모 자식의 관계로 표현
  2. 망형 데이터 모델 : 그래프 구조. 상위와 하위가 M : N관계. 개체간 관계를 오너와 멤버의 관계로 표현
  3. 객체 지향형 데이터모델(OODM) : 객체 개념을 데이터베이스에 도입함.
  4. 객체 지향형 데이터 베이스(OODB) : 객체지향 데이터 모델의 개념을 반영하여 실세계에 존재하는 객체를 표현하고 관리하는 데이터 베이스 기술
  5. 관계형 데이터 모델 : 표(table)을 이용하여 데이터 상호관계를 정의하는 DB구조

튜플의 수 : 카디널리티, 기수, 대응수

속성의 수 : 디그리(degree), 차수

도메인 : 하나의 애트리뷰트가 위하라 수 있는 같은 타입의 원자값들의 집합

후보키 : 기본키로 사용할수 있는 속성들. 유일성, 최소성

대체키 : 기본키를 제외한 나머지 후보키

슈퍼키 : 속성들의 집합으로 구성된 키. 유일성 만족 최소성 X

무결성

무결성 : 데이터 베이스에 저장된 데이터 값과 그것이 표현하는 현실 세계의 실제값이 일치하는 정확성
널 무결성 : 특정 속성값이 NULL이 될 수 없음

고유(Unique) 무결성 : 각 튜플이 갖는 값들이 서로 달라야 한다는 규정

도메인 무결성 : 특정 속성의 값이, 속성이 정의된 도메인에 속한 값이 되어야 한다는 규정

키 무결성 : 하나의 테이블에는 적어도 하나의 키가 존재해야함

관계 무결성 : 관계에 대한 적절서 여부 지정

    • Brute-force 혹은 Henerate-and-Test 기법이라고 불림

    • 모든 경우의 수를 생성하고 테스트하기 때문에 수행 속도는 느리지만, 해답을 찾아내지 못할 확률이 적음
    • 주어진 문제를 풀 때, 우선 완전 검색으로 접근하여 해답을 도출한 후, 성능 개선을 위해 다른 알고리즘을 사용하고 해답을 확인하는 것이 바람직하다

    • 서로 다른 n개 중 r개를 택하는 순열은 nPr
    • nPr은 다음과 같은 식이 성립 : nPr = n*(n-1)(n-2) … * (n-r+1)

우분투에서 python 가상환경 구축하기

제 우분투 버전은 18.04LTS이며 ZShell을 사용하였습니다.
IDE는 Pycham을 기준으로 작성하였습니다.

  1. 파이썬 프로젝트를 생성합니다.

  2. 하단의 터미널을 켜서 pip3 install virtualenv 를 치고 설치해 줍니다

    • 저는 파이썬 기본설정이 Python2이기 때문에 pip3라고 하였습니다. 기본설정이 Python3이면 pip라고 하셔도 무방합니다.

  3. 설치 후 잘 설치가 되었는지 확인하기 위해 virtualenv --version을 입력합니다.

  4. 위와 같이 버전이 뜨면 정상적으로 설치가 되었습니다. 그 다음 가상환경을 만들어 보겠습니다. virtualevn [가상환경명] 을 적어줍니다


    적고 실행시키면 목록이 뜨게 됩니다.

  5. 그럼 이제 가상환경을 실행해 보겠습니다.
    source [가상환경명]/bin/activate를 실행합니다.

    실행 하고나면 이런식으로 맨 왼쪽에 [가상환경명]이 뜨게 되며 적용이 된것을 확인할 수 있습니다.
    이 상태에서 **pip3 install [패키지명]**을 통해 패키지를 설치하시면 됩니다.

    • [패키지명]==[버전]을 하게되면 원하는 버전으로 설치가 가능합니다.

  6. 가상환경을 종료하시려면 deactivate를 실행해 줍니다.

2019-04-21 localhost와 127.0.0.1의 차이점

2019/04/21 카테고리 네트워크

127.0.0.1은 무엇인가?

127.0.0.1은 IPv4 및 IPv6에서 자기 자신을 가리키기 위한 목적으로 쓰기 위해 예약된 IP이다. 모든 컴퓨터는 이 주소를 자신의 주소로 사용하지만 실제 IP 주소처럼 다른 장치와 통신 할 수는 없다.

  • Ex) 컴퓨터에서 실행중인 웹 서버는 127.0.0.1을 가리킬 수 있으므로 페이지를 배포하기 전에 로컬에서 실행하고 테스트 할 수 있다.

LocalHost는 무엇인가?

localhost은 자기 자신을 가리키는 IP의 도메인 네임으로써 알기 쉽게 해놓은 것이다. 그러므로 localhost로 접속을 하게되면 DNS의 테이블과 비교를 해야하는 번거로움이 있다 그로 인하여 127.0.0.1로 직접 접속하는것 보다 느리지만 기억하기는 쉽다.
그리고 localhost는 hosts 파일을 통해 변경될 수 있다

루프백

호스트 자신을 가리키는 주소이다 127.0.0.1을 사용한다. 루프백 주소로 네트워크에 직접 연결하지 않은 상태에서 빠르게 프로그램의 테스트를 할 수 있다. 또한 루프백 IP를 갖는 패킷은 외부 네트워크로 전송할수 없다

Django 프로젝트 구조 살펴보기

    • init.py : 파이썬 2.x대 버전과의 호환을 위해 만들어진 비어있는 파일이다. 지워도 무방하다
    • settings.py : 프로젝트 설정에 관한 다양한 내용이 들어있다.
    • urls.py : 특정 기능을 수행하기 위해 접속하는 주소를 url이라 하고 이를 설정해주는데, 그 설정을 이 파일에 기록함
    • wsgi.py : 웹 서비스를 실행하기 위한 wsgi 관련 내용이 들어있음

  2. venv 폴더 : 프로젝트 구동에 필요한 가상환경이 들어있음

  3. db.sqlite3 : SQLite3 DB 파일이다. 다른 DB를 이용할시 필요없다

장고 기본 명령들

  1. django-admin startproject : 장고 프로젝트를 만드는 명령이다. 웹 서비스를 만들 때 마다 한번 실행한다.
  2. startapp : 프로젝트에 기능 단위인 앱을 새로 만들때 사용
  3. makemigrations : 어플리케이션에 변경 사항을 추적해 DB에 적용할 내용을 정리함. 보통 앱 안에 있는 모델(model)의 변경 사항이 있을 때 주로 사용함.
  4. sqlmigrate: 실행할 SQL 명령문을 출력한다
  5. migrate : 실제 변경사항을 DB에 반영한다.
  6. showmigrations : 프로젝트의 DB 변경사항 목록과 상태를 출력한다.
  7. runserver : 테스트 서버를 실행한다.
  8. dumpdata : 현재 DB의 내용을 백업할 때 사용한다.
  9. loaddata : 백업 파일에서 DB로 내용을 복구 할 때 사용한다
  10. flush : DB 테이블은 그대로 두고 테이블의 내용만 전부 삭제한다
  11. shell : 장고 쉘(Shell)을 실행한다
  12. dbshell : DB에 직접 접근 할 수 있는 쉘을 실행한다
  13. createsuperuser : 관리자 계정을 생성한다
  14. changepassword : 계정의 비밀번호를 변경할 수 있다
