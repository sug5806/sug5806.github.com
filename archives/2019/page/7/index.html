Classful addressing

  • IP주소의 구성 : Network 주소. Network주소. Host주소. Host주소

C 클래스 : 소규모 네트워크에 적용

  • 총 2,097,150개의 네트워크를 만들 수 있으며 각 네트워크마다 254개의 노드를 연결할 수 있다
  • IP주소의 구성 : Network 주소. Network주소. Network주소. Host주소

Init으로 시작하는 방법

레포 이름과 폴더 이름이랑 같지 않으면 나중에 헷갈릴수 있으므로 똑같이 하는게 좋다

  1. 본인의 git 저장소로 쓰고싶은 로컬 디렉토리로 이동 한다음 git init을 한다

  2. git config --global user.name "username"을 입력한다

  3. git config --global user.email "email"을 입력한다

    • git 설정을 할때 git config --global을 하면 전체에 설정이 되고 global을 빼면 해당 레포?에서만 지정이 된다.

  4. git remote

    • get-url 별명 : 별명이 가지고 있는 주소를 알 수 있다
    • 별명 주소 : 별명으로 주소의 리모트 레포를 이용 하겠다 반드시 별명이 있어야 함

  5. git add

    • git add 파일이름을 하여 staing으로 올려준다.

  6. commit을 할때는 동작을 하는 코드만 commit을 해야한다

    • -u를 붙임으로써 같은 브랜치라고 알려준다(브랜치 당 최초 1번만 하면 된다)
      • (git commit -u 별명 master)
    • commit을 할때 기능에 따른 말머리를 추가하는것이 좋다
    • commit 메시지를 작성할때는 현재형으로 작성하는것이 좋다
    • commit 메시지를 적을때는 제목으로 무엇을 변경했는지 알아볼수 있게 작성하는게 좋다
    • collection (list, dictionary, tuple 등)을 사용할때는 나열식으로 하는것 보다는 값마다 한줄 한줄씩 하는게 좋다
    • collection을 사용할때는 마지막 원소에 ,(콤마)를 적어주면 깃에서 추가된 내용만 추가가 된다.
    • commit을 할때는 동작하는 최소 단위로 해주는 것이 좋다(function단위도 좋은 방법)

Clone으로 시작하는 방법

  1. 레포로 사용하고싶은 디렉토리로 이동한다음 git clone "github 주소"를 하면 자동으로 복사를 해서 만들어 준다.

private은 무료이나 통계기능이 잠겨있다 -> 돈을 내고 사용해야 풀림

gitignore : 무시하고 싶을때 포함하고 싶은 파일 확장자를 입력하면 push를 할때 깃에 올라가지 않게된다

index가 있으므로써 commit을 개별로 할 수 있고 local repo가 있으므로 offline 개발이 가능해진다

2019-03-29 컴파일러와 인터프리터의 차이점

2019/03/29 카테고리 컴파일러이론

둘 사이의 가장 단순한 차이점은 입력시점과 실행시점의 차이이다. 컴파일러 언어는 컴파일 시간과 실행시간이 따로 있어 소스코드는 컴파일 시간에 들어오고 런타임때 입력데이터가 들어온다. 그러나 인터프리터 언어는 컴파일 시간이없고 실행 시간만 있으며 소스코드와 입력데이터가 같은 시점에 들어온다는 점이다.

컴파일러

초창기의 컴퓨터는 기계어로 프로그래밍을 했다. 그러나 기계어로 프로그래밍을 하기는 너무 어려워 이를 보다 편하게 사용하기 위해 방법을 생각했다.

  1. 일정한 형식으로 작성된 문서를 기계어로 자동 번역하는 프로그램 A를 만든다.
  2. 이후 프로그램을 만들때마다 A가 번역할 수 있도록 일정한 형식으로 문서를 작성한다.
  3. 문서 작성이 완료되면 프로그램 A를 먼저 실행하고 파일을 넘겨서 A가 자동으로 번역해준 기계어 파일을 얻는다.

그리고 여기서 사용되는 프로그램 A를 컴파일러(compiler) 라 하고, 이때 작성한 일정한 형식으로 작성한 문서를 소스파일(source file) or 소스코드(source code), 그리고 이를 번역하는 행동을 컴파일(compile) 이라 한다.

그리고 시간이 흘러 파일 하나에 모든 소스코드를 작성하는 방식은 불편하다고 느껴 여러 파일에 나눠 작성을 하고 이렇게 나뉘어 컴파일된 파일을 모아 하나의 실행가능한 파일을 만들면 이것을 링크(link) 했다고 하고, 이때 사용되는 프로그램을 링커(linker) 라고 한다.
이것을 종합하면

  1. 소스 코드를 작성하고 파일로 저장한다.
  2. 저장된 파일을 컴파일러를 이용하여 컴파일한다. 목적 파일이 생성된다.
  3. 생성된 목적파일을 링크하여 링커를 통해 실행 가능한 목적 파일이 생성된다.

컴파일러와 링커 모두 목적 파일을 생성하나 컴파일러는 실행이 불가능하고 링커는 실행 가능한 목적파일을 생성한다는 점이다.

컴파일과 링크 과정을 합쳐 빌드(build) 라고 하고, 이때 사용되는 프로그램을 빌더(builder) 라고 한다.


사진 출처 https://github.com/ythwork/CS/blob/master/interpreter/interpreter.pdf

그러나 컴파일러 언어는 단점이 있는데 기계어는 기계마다 그 내용이 다르기 때문에 똑같은 프로그램 일지라도 기계가 바뀔 때마다 해당 기계에 맞는 기계어로 다시 번역을 해야한다. 그러나 소스코드와 기계어 사이에 중간 단계의 언어가 있어 소스코드가 중간 단계의 언어로 먼저 번역된 다음, 번역된 중간 단계의 언어가 기계어로 번역된다면 위와같은 문제점을 해결할수 있지 않을까?

인터프리터

인터프리터(Interpreter) 란 컴파일 과정을 거치지 않고 소스 코드를 바로 해석하여 결과를 출력하는, 소스 코드 실행기 프로그램을 말한다. 둘 사이의 차이점을 알아보자

  1. 먼저 컴파일러는 컴파일을 하고나서 그 즉시 실행을 하지는 않는다. 그러나 인터프리터는 실행기 이므로 번역과 분석이 끝나면 바로 프로그램을 실행하여 결과를 내놓는다.
  2. 컴파일러는 기계어로 번역을 한 후 목적 파일을 생성하는 반면 인터프리터는 그렇지 않다는 점이다.
  3. 컴파일러는 컴파일하는데 시간이 걸리지만 끝나고 나면 분석이 완전히 되었으므로 다시 분석할 필요없이 바로 실행을 하면되지만 인터프리터는 실행을 할때마다 매번 분석을 진행하므로 컴파일러보다 속도가 느릴 수 밖에 없다

2019-03-22 OOP

2019/03/22 카테고리 기타

Procedural vs Object-Oriented

  • procedural -> 함수를 이용해서 추상화를 했다
  • Object-Oriented -> 객체를 통해 추상화를 했다
    • 객체 단위로 프로그래밍을 한다
    • 객체(Object) -> ‘관련있는’ 변수(상태정보)와 ‘관련있는’ 함수를 한곳에 모아놓은 ‘곳’ (곳 = 메모리) -> bundling
      • attribute(속성)
        • instance member(변수)
          • 각각의 객체가 가지는 값
        • instance method(함수)
          • 객체가 할 수 있는 행동, 기능

message passing

  • 다른 객체(other)의 상태정보(멤버)를 변경할 때는 반드시 상대 객체가 가진 메서드를 이용한다

클래스는 함수의 모임

객체와 인스턴스의 차이
객체와 인스턴스의 메모리는 같음
인스턴스=> Ex) me라는 객체는 Account라는 클래스의 인스턴스

OOP

  1. 캡슐화(Encapsulation)

    1. 관련있는 멤버(변수) 와 메서드(함수)를 하나의 단위(대부분 클래스)로 묶는 것
      1. 클래스를 제외하고도 하나의 단위로 묶을수 있다면 OOP라고 함
        1. Ex) C에서의 구조체
        2. the bundling of data with methods
      2. 정보 은닉을 포함함

  2. 정보은닉( 파이썬은 완벽한 정보은닉을 지원 안함) -> C++에서 클래스 만들면서 private선언

    1. 어떤 멤버를 공개 / 비공개
    2. 어떤 메서드를 공개 / 비공개

  3. 다형성

    1. 상속
      • 상속을 쓰지 말자
      • 상속은 OOP의 정수이다
        • 상속은 쓰되 다중상속은 하지말자
      • 멤버변수에는 절대 직접접근 금지
      • 객체의 멤버에 접근 -> getter / 수정 -> setter 을 수행할때는 method function을 이용해야함
      • 내부에서는 멤버처럼 쓰는것처럼 하면서 property를 사용하여 정보은닉을 함
      • 인스턴스 멤버를 만든다고하면 지금 당장 쓰지 않아도 되도록이면 생성자에서 만들어 주는게 좋음
      • 멤버는 어디에서든지 생성될 수 있음
    2. Relations between classes
      • choose either inheritance or composition (언제 상속으로하고 언제 컴포지션으로 하는가)
        • IS - A
          • A laptop IS - A computer / ~은 ~의 한 종류이다
          • Inheritance (상속)
          • 조건 1 : 모든 멤버와 모든 메서드의 중복이 되어야함
          • 재사용성(reusability) -> 재사용성 만으로 IS -A 상속을 하면 절대 안되고 조건을 모두 충족해야함!
        • HAS - A
          • A policeman has a gun / ~이 ~을 가지고 있다. or 포함하고 있다
          • Composition(합성) -> ◆(In UML)
            • same life cycle = 강하게 연결되어 있다
          • Aggregation(통합) -> ◇(In UML)
            • NOT same life cycle = weak coupling
    3. method overriding (메소드 오버라이딩) -> 상속이 필수
      • polymophism (다형성)
        • 상속 관계에 있는 다양한 클래스의 객체에서 같은 이름의 메서드를 호출할 때, 각 객체가 서로 다르게 구현된 메서드를 호출함으로써 서로 다른 행동, 기능, 결과를 가져오는 것
    4. abstract class (추상 클래스) = 자바에서의 인터페이스
      • 인스턴스를 만들 수 없다.
      • 하나 이상의 abstract method (추상 메소드)
        • 몸체(body)가 없는 함수 -> pass
      • 파생 클래스에서 반드시 overriding을 해야함 -> 구현을 강제하도록 한다

CPU :

  • 데이터 버스 : 데이터가 이동하는 통로(양방향)
  • 주소 버스 : 주소 이동하는 통로(단방향)
  • CU : 실제로 명령을 실행하는 주체 (총감독)-> IR에 있는 명령어를 읽어 실행함
  • PC(Program Counter) : 다음에 실행될 인스트력션(기계어)이 메모리(RAM)에서 저장된 곳의 주소값을 저장하는 레지스터
    • 기계어 : 어셈블리어(기계어를 1:1로 대응해서 사람이 읽을 수 있게 바꾼것)
  • IR(Instrcution Register : 현재 실행 중인 인스터럭션(기계어) 자체가 저장된 레지스터
    • 다음에 실행할 명령어를 가져오기 위해 제어 버스(Control Bus)에는 READ 모드를 전달하고, 주소 버스에는 다음에 실행할 명령어인 add ax, bx가 저장되어 있는 주소인 0x2를 전달합니다 주소 0x2번지에서 명령어 add ax, bx를 데이터 버스를 통해 IR로 전달합니다
  • ALU = 사칙연산과 논리계산
    • 컨트롤 버스의 read라인을 통해 모드를 전달하고 메모리의 스택프레임 안에있는 변수의 주소를 통해 값을 읽고 데이터 버스를 통해 읽어와 AX로 저장한후 ALU로 계산을 하고 다시 AX에 저장을 한뒤에 메모리에 전달하여 저장함
  • 범용레지스터 : 데이터를 일시적으로 저장
  • 클럭 : 클럭에 맞춰 인스트럭션의 전달을 완료하거나 전달하고있는 과정

instruction cycle

  • fetch : PC가 가리키고 있던 익스트럭션을 메모리에서 CPU의 IR로 가져오는 과정
      1. 다음에 실행할 명령어를 가져오기 위해 제어버스에 READ모드를 전달하고, 주소버스에는 다음에 실행할 명령어가 저장되어 있는 주소를 전달한다.
      1. 해당 주소에 있는 명령어를 데이터버스를 통해 IR로 전달한다.
    • MMU를 통해 PC에있는 메모리주소를 convert하고 MAR로 전달
  • decode : CU가 IR을 분석, 해석하는 과정
  • execute : 연산을 하는것
  • wirte back : 최종적으로 연산한것을 메모리에 저장하는 과정

하드웨어적) 한번에 보낼 수 있는 데이터의 양 ___ CPU와 메모리가 연결되어있는 와이어 갯수가 곧 bit (ex 64bit 컴퓨터는 와이어가 64개)
32bit --> 232 / 64 bit --> 264

주기 : 인스트럭션이 실행되거나 실행되고있는 시간
CPI (Clock cycles per instruction) : Load계열의 경우(CPU내부말고 외부를 거쳐야 하는 명령어)는 Instruction당 clock수가 3정도됨 add같은 명령어는 1임 Ex) 더 짧은데 실행시간은 더 길다 = Line당 CPI가 높다

로우레벨은 오로지 어셈블리어 하나 뿐 : 어셈블리어는 기계어로 1:1로 대응됨, 어셈블리어를 하려면 레지스터 이름까지 다 알아야함, CPU가 다르면 다시 프로그래밍 해야함, 하드웨어에 의존적임

ESP(Stack Pointer) : 스택프레임을 만들고 스택세그먼트의 맨위를 가리킨다
FP(Frame Pointer) = BP(Base Pointer) : 스택 frame의 기준(가운데)을 가리킨다, 변수에 접근할때 기준점
EBP(extended base pointer) : 함수의 기준점을 가리키고있음 -> main의 기준점을 가리키고있다가 add함수를 호출하면 main함수의 기준점 주소를 add함수 스택에 쌓음

  • EBP를 기준으로 ±를 하면 변수에 접근이 가능함
    ESP -> ESP - 지역변수 갯수 만큼 -를 시켜 공간을 미리 확보함 : 스택은 주소공간에서 맨 아래부분(주소가 큰부분 :0xFFFFFFFF)부터 할당되므로 -를 해야 확장 개념임
  • 미리 확보해놓는 이유 : 스택프레임이란 함수가 실행되면 함수의 지역변수를 미리 확보해야 하나 그렇지 않으면 스택 프레임이란 개념이 깨짐

함수내에서 다른 함수를 불러오기직전에 함수호출 다음의 주소(return address)를 스택에 쌓아둠

Memory :

  • Locality
    • 공간 지역성 : 한번 접근했던 메모리가 이전에 접근한 메모리의 근처일 확률이 높다.
    • 시간 지역성 : 한번 접근했던 메모리에 계속 접근한다. result += e에서 result
  • Cache
    • cache line = 64 Byte ~ 128 Byte(메모리에서 가져올때 뭉텅이로 가져온다)
    • cache miss : CPU가 요청했을때 cache에 없어 메모리에서 가져오는것
    • cache hit : CPU가 요청했을때 cache에 있어 바로 보내주는것
  • Segment: Ex) 윈도우는 메모리가 4G라면 유저영역을 2G (유저영역또한 Code, Data, Heap, Stack으로 나눠지고 커널영역도 동일) 커널 영역도 2G , 리눅스는 3G
    • 가상주소공간(virtual address space) : OS가 프로세스가 시작될때 할당해주는 가상 메모리공간
    • CODE : 코드가 변환된 인스터럭션(기계어)이 들어감
    • DATA : 전역변수(생성-> 프로세스가 시작될때 / 종료-> 프로세스가 종료될때)
      • Data : 선언과 동시에 할당을 할 경우 들어감
      • BSS : 그냥 선언만 할 경우 BSS에 들어감
    • HEAP : 생성 -> 프로그래머가 원할 때(malloc) / 종료 -> 프로그래머가 원할 때(free)
      • Heap의 단점 :
          1. 메모리 누수
          1. 느림(malloc) -> 더블 링크드 리스트로 구성되어있어 저장할 공간을 찾을때까지 뒤지고 다녀 느림
          1. 메모리 단편화(memory fragmentation)
      • 제일 처음 잡히는 HEAP을 default Heap이라 함(OS가 줌)
      • 처음부터 내가 원하는 크기만큼 받고 프로그래머가 입맛에 맞게 분할하여 사용하는 기법이 다이나믹 힙(Dynamic Heap)기법이다
        • Ex) 데이터를 띄엄띄엄 저장되는것을 방지하여 한쪽으로 몰아넣기
    • STACK : 지역변수(생성 -> 함수가 호출될때 / 종료 -> 함수가 종료될때)
      • 메모리 할당에서 부터 SP가 가리키고 있으므로 찾을 필요없이 차곡차곡 쌓으면 되므로 Heap보다 빠름
      • default값으로 1MB로 잡혀있음

Gabage collection

  • 언어 차원에서 Heap 메모리 관리를 함
  • reference couner : 가리키는게 아무것도 없으면 알아서 삭제함(파이썬이 이렇게 구현함)

Virtual memory (RAM + Page File) / 리눅스(RAM + Swap)

  • 4096 Byte(4KB) 블록 하나를 페이지라고 함 (32bit 컴퓨터)
  • 232 / 212(=4K) = 220
  • page, page number, offset
    • Ex) 총 2000byte , page당 100byte, 총 페이지수 20개
    • page#0 = 0~99번지, page#1 = 100~ 199번지
      • 번지수를 100의 자리수로 나누면 그게 곧 페이지 넘버
        • Ex) 140 = 1은 페이지넘버 / 40은 offset 각 페이지의 시작주소에서 40바이트만큼 떨어져있는 곳 -> 140 = 페이지넘버 + offset
        • 140번지의 메모리 주소를 봤더니 1111a/bab라면 앞에있는 20bit가 page넘버 / 뒤에있는 12bit가 offset
  • Page File : 디스크의 일정 용량을 RAM처럼 쓰겠다 -> 일정 용량을 Page File이라 함, victim page의 집합
  • page out : 새로운 데이터를 올리기위해 잘 돌고있는 친구를 page file로 내리고 내려진 친구를 victim page라 부름 하고 이러한 행위
  • page in : page file에 있는 요청된 페이지를 올리는 행위
  • 쓰레싱 : CPU의 요청사항이 너무많아 Page in, Page out이 무한적으로 일어나는 행위

실제 Main Memory

  • frame = 가상메모리의 page를 나누는 것과 동일, frame# = page# 가상메모리와 똑같음

CPU가 명령할때

  • page 넘버를 frame넘버로 바꿔 MAR(Memory Address Register)에 전달해주고 전달받은 주소로 가서 데이터를 가져옴

MMU : PC가 가리키고있는 logical address(가상메모리 주소)를 runtime동안 page table를 참조하여 physical address(실제 메모리)로 변환함 (CPU내부에 있음)

  • MMU는 PTBR에 저장된 페이지 테이블의 시작주소를 참조하고 PC에서 가상페이지넘버 상위 20비트를 참조, 페이지 테이블에서 페이지넘버를 찾고 이에 상응하는 프레임넘버를 가져와 PC의 오프셋 비트(12bit)와 합쳐 물리주소를 만듬. 이 물리주소를 MAR에 저장한다음 CPU는 이 레지스터의 주소 값을 읽어와 메인 메모리에서 인스트럭션(명령어)를 가져오고(fetch) 실행함(execute)
    PTBR : 페이지 테이블의 시작주소를 가리키는 레지스터(CPU내부에 있음)

Page Table : OS가 프로세스를 시작하면 4G를 할당 해준다(VAS), 프로세스마다 페이지 테이블을 main memory에 만들고 page table의 첫주소를 가르쳐준다.

  • 이러한 페이지 테이블을 만드는거 자체를 4G를 할당해준다라고 친다.
  • 지금 당장 필요 할것같은 코드영역, 전역변수등을 메인메모리에 먼저 업로드 -> preparing기법

demand page : CPU가 페이지를 요청할때 올리는것

현대 운영체재는 세그먼트 + 페이징기법을 같이 씀

  • CPU가 요청하였을때 (Heap에 데이터를 저장할때 순서대로가 아닌 서로 떨어진 빈공간에 넣으므로) 매번 새로운 페이지를 가져올 확률이 생김

문자와 문자열, 함수전달 방식

  1. 문자, 문자열 -> character set, ASCII( -> UNI-CODE(테이블로 이루어져 있음)-> UTF-8, UTF-16, UTF-32)

    • character set -> 문자 집합(Ex. 라틴문자)

    • character encoding -> 문자 인코딩(Ex. 모스 부호)

    • code point -> 문자 하나에 정수 하나를 매핑해둠(Ex. 아스키 코드표(7bit로 모두 표현가능)

    • UniCode -> 2byte 기반으로 만듬(2byte 테이블을 17개 만들어놓았고 현재 실생활에서 쓰는건 2개(4개 등록되어있음))

      • 216개만큼의 문자가 매핑되어있음 BMP -> BMP까지는 2byte면 다 표현이 가능
      • 아스키는 코드포인트 자체 그대로 메모리에 저장함
      • 유니코드는 메모리에 저장할때 utf-8, utf-16, utf-32로 나눠서 저장
      • UTF-16의 경우 BMP에 해당하는 문자는 그대로 전송(2byte), 2차평면 이상부터는 4byte로보내는 대신 앞을 0으로 다 채워서 보냄
      • UTF-32는 처음부터 4byte로 바꿔서 보냄
      • UTF-8 -> 가변길이 인코딩, 한글은 3byte로 인식, 엔디언 변환을 무시(Endianness)하고 그대로 저장, 전송을 함

    • 사람들이 인식하는 것은 빅엔디언형식이고 컴퓨터는 대부분 Intel CPU를 사용하고 intel은 리틀엔디언 방식으로 저장함

U+AC00 을 메모리에 저장할때는 리틀엔디언 방식을 통해 00 AC로 저장함
네트워크를 통해 데이터를 보낼때는 빅엔디언으로 보냄
실제로 네트워크를 통해 전달되는 방식으로 변환되는것이 유니코드인코딩방식?

  1. 함수를 호출할때 인자 전달 방식 ->> call by value, call by reference -> C언어 call by object reference(call by assignment) -> 파이썬
    • call by value : 값에 의한 전달 -> 함수 스택프레임 안에서 값이 바뀌어도 밖의 스택프레임에는 영향을 주지않는것을 보장함
    • call by reference : 주소에 의한 전달 -> 함수 스택프레임 안에서 주소를 통해 바깥 스택프레임에 있는 변수에 영향을주어 값을 변화시킴
