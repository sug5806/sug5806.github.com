Amazon CLI를 사용하여 사용자 등록하기

Django로 Amazon 서비스를 연동하다 보면 settings.py에 아마존 aws_access_key_id와 aws_secret_access_key를 입력해야 하는데 github 같은 곳에 실수로 올리게 되면 해커들이 도용을 하게됩니다. 이것 방지하기위해 AWS CLI를 통해 등록하면 settings.py에 더이상 access_key를 적지 않아도 됩니다.

  1. 먼저 아마존에 가입을하고 IAM항목으로 들어갑니다.

  2. 그룹을 누르고 새로운 그룹생성을 합니다.

  3. 그룹이름을 적고 우측하단에 다음단계를 누릅니다

  1. 본인이 이용하실 서비스에 대한 정책을 선택하고 우측 하단에 다음단계로 진행합니다. 저는 S3를 이용할것이므로 S3FullAccess를 선택하였습니다.
  1. 선택하신것을 확인하시고 우측 하단에 그룹선택을 눌러줍니다.
  1. 이제 그룹에 사용자를 추가시켜줘야 합니다. 사용자 탭을눌러 사용자 추가를 해줍니다.
  1. 사용자 이름을 적어주시고 우리는 CLI를 통해 등록을 할것이므로 프로그래밍 방식 액세스에 체크하도록 합니다. 그리고 우측 하단의 다음으로 넘어갑니다
  1. 방금 만든 그룹이 보이실텐데 추가하고 싶은 그룹을 체크박스 하신뒤 우측 하단의 다음버튼을 눌러줍니다
  1. 태그 추가는 지금당장 할일이 없으므로 생략하도록 합니다
  1. 이제 그동안 설정한 값들을 확인할 수 있는 화면입니다. 확인후 사용자 만들기를 누릅니다.
  1. 제일 중요한 화면입니다. 이 화면을 벗어나게되면 확인할 수가 없습니다. 그러므로 .csv 다운로드를 통해 등록한 사용자의 ID와 시크릿키를 다운받습니다. 절대 다른사람에게 노출이 되면 안됩니다.
  1. 이제 django에서 사용자를 등록하도록 하겠습니다.
    pip install awscli를 입력하여 설치해줍니다
  1. 설치가 다 되면 aws configure을 통해 다운받으신 csv파일의 ID와 Key를 입력하시면 됩니다. region은 서울이니 ap-northeast-2, output format은 json, text, table중 하나를 입력해주면 됩니다.
  1. 등록을 마치면 더이상 settings.py에 ID와 Key를 입력하지 않아도 자동으로 본인 root디렉토리에 .aws디렉토리 내부의 파일을 읽어 사용할 수 있습니다.

커스텀 유저모델 만들기

커스텀 유저 모델은 기존의 유저 모델에 추가 필드나 기능을 사용하고 싶거나 이전에 존재하는 유저 데이터를 마이그레이션 해서 사용하고 싶을 때 선택하는 기법입니다. 커스텀 유저 모델을 3가지가 존재합니다.

  1. 기존의 유저 모델을 확장하여 추가 정보만 입력받는 방법

  2. AbstractBaseUser를 사용해서 기본 기능을 제외한 모든 기능과 필드를 새로 만드는법

  3. AbstractUser를 사용해서 추가 필드를 포함한 유저 모델을 만드는 방법

세가지 방법중 가장 편리한 방법은 3번째 방법인 AbstractUser를 사용하는 방법입니다.
이 방법의 주의할 점은 기존 유저 정보를 사용할 수 없기 때문에 프로젝트를 시작할 때 커스텀 유저 모델을 만들고 관리자 계정 생성을 진행해야 한다는 것입니다.

  1. 새 프로젝트를 만들고 account 앱을 만듭니다.

  2. settings.py의 INSTALLED_APPS에 accounts 앱을 추가해줍니다.

  3. 모델에서 AbstractUser 클래스를 상속받아 새로운 User 모델을 작성합니다. 모델의 이름은 User일 필요는 없습니다. User 모델에 원하는 추가 필드를 작성합니다.

  4. 새로운 유저 모델을 만들면 settings.py에 모델을 설정해야 합니다.
    AUTH_USER_MODEL이라는 변수를 만들고 값을 설정합니다.

  1. 새로 만든 유저 모델의 DB 생성을 위해서 makemigrations 명령을 실행하겠습니다. 그런데 유저 모델에는 이미지 필드가 있으므로 Pillow 모듈을 먼저 설치합니다.


  1. 관리자 계정을 생성합니다

  2. 서버를 실행하고 관리자 페이지에 들어가서 확인해 봅시다.

  3. 관리자 페이지에 로그인을 하면 목록에 유저 모델이 나타나지 않습니다. 유저 모델을 확장하여 새로 만들었기 때문에 새로 등록을 해주어야 합니다. admin.py에 유저 모델을 추가해 봅시다.

  4. 관리자 페이지를 새로 고침하면 유저 모델이 나타나는 것을 확인할 수 있습니다.

  5. 유저 모델에 등록되어 있는 관리자 계정의 수정 페이지로 이동합니다. 우리가 만났던 기존의 사용자 관리 화면과는 많이 다릅니다. 기존에 사용했던 페이지와 흡사하게 변경해보겠습니다.

  1. admin.py에 UserAdmin 클래스를 옵션으로 등록해줍니다. 우리가 admin.py에서 옵션 클래스를 만들어 등록하면 관리자 페이지의 목록을 커스터마이징 했던것처럼 유저 모델도 이 옵션클래스가 필요한데, 기존에 있던 Useradmin 모델이 이 옵션 클래스 역할을 하고 있습니다.

  1. 그런데 우리가 만든 추가 필드가 나타나지 않습니다. 이것을 나타나게 하려면 Option 클래스를 새로 만들어 등록해줘야 합니다. 기존의 UserAdmin 클래스에는 fieldsets와 add_fieldsets가 존재하는데 fieldsets는 상세 화면에서 출력될 수정 폼 설정이고 add_fieldsets는 추가 화면에 출력될 입력 폼의 설정값 입니다. 두가지에 모두 추가 필드를 입력하겠습니다.

  2. 새로고침해서 화면을 확인해 봅시다. Profile과 Message 값이 잘 출력되는 것을 확인할 수 있습니다.

개발 환경과 배포 환경 상이하다

컨테이터 서비스 도커 : 가상환경 구축(운영체제 환경까지 어느 정도 지원)
모토 : 개발환경을 그대로 배포환경까지 이어가겠다.

  1. LCX 시스템 - 도커는 리눅스에서만 동작한다.
  2. runC - 리눅스, 맥, 윈도우 모두 가능

도커 시스템은 무조건 리눅스이다

장점 : 개발환경을 그대로 감싸서 배포까지 한번에
단점 : 배포 환경 그대로 개발 환경에서 사용해야 한다.

도커 이미지를 만드는 절차

  1. 기본 베이스 이미지를 선택한다. - 도커 허브(각종 배포용 이미지가 있다) 기본 리눅스 이미즐 선택
  2. 컨테이너 구동(개발용, 배포용)
    2-1) 개발 환경 셋팅 - Docker file 이라는 곳에 명령행을 기록
    2-2) 변경사항이 남아있는 이미지를 생성
  3. 이미지를 리포지터리에 업로드
  4. 배포
    4-1) 도커 - 리눅스시스템, 서버 시스템에서 도커이미지를 받아서 여러개의 컨테이너를 구동
    4-2) 엘라스틱 빈즈토크 - 도커용 시스템
  1. 개발+배포용 도커이미지 만들고
  2. 해당 이미지를 사용해서 장고 어플리케이션 구동
  3. 개인 리포지터리에 업로드

코드를 업데이트 하는 방법

  1. Dockerfile에 ADD 명령을 통해 넣기
  2. container를 마운팅해서 넣는 방법 : -v 내 프로젝트 경로 : container의 마운팅시킬 디렉토리의 경로
