2019-03-22 OOP

2019/03/22 카테고리 기타

Procedural vs Object-Oriented

  • procedural -> 함수를 이용해서 추상화를 했다
  • Object-Oriented -> 객체를 통해 추상화를 했다
    • 객체 단위로 프로그래밍을 한다
    • 객체(Object) -> ‘관련있는’ 변수(상태정보)와 ‘관련있는’ 함수를 한곳에 모아놓은 ‘곳’ (곳 = 메모리) -> bundling
      • attribute(속성)
        • instance member(변수)
          • 각각의 객체가 가지는 값
        • instance method(함수)
          • 객체가 할 수 있는 행동, 기능

message passing

  • 다른 객체(other)의 상태정보(멤버)를 변경할 때는 반드시 상대 객체가 가진 메서드를 이용한다

클래스는 함수의 모임

객체와 인스턴스의 차이
객체와 인스턴스의 메모리는 같음
인스턴스=> Ex) me라는 객체는 Account라는 클래스의 인스턴스

OOP

  1. 캡슐화(Encapsulation)

    1. 관련있는 멤버(변수) 와 메서드(함수)를 하나의 단위(대부분 클래스)로 묶는 것
      1. 클래스를 제외하고도 하나의 단위로 묶을수 있다면 OOP라고 함
        1. Ex) C에서의 구조체
        2. the bundling of data with methods
      2. 정보 은닉을 포함함

  2. 정보은닉( 파이썬은 완벽한 정보은닉을 지원 안함) -> C++에서 클래스 만들면서 private선언

    1. 어떤 멤버를 공개 / 비공개
    2. 어떤 메서드를 공개 / 비공개

  3. 다형성

    1. 상속
      • 상속을 쓰지 말자
      • 상속은 OOP의 정수이다
        • 상속은 쓰되 다중상속은 하지말자
      • 멤버변수에는 절대 직접접근 금지
      • 객체의 멤버에 접근 -> getter / 수정 -> setter 을 수행할때는 method function을 이용해야함
      • 내부에서는 멤버처럼 쓰는것처럼 하면서 property를 사용하여 정보은닉을 함
      • 인스턴스 멤버를 만든다고하면 지금 당장 쓰지 않아도 되도록이면 생성자에서 만들어 주는게 좋음
      • 멤버는 어디에서든지 생성될 수 있음
    2. Relations between classes
      • choose either inheritance or composition (언제 상속으로하고 언제 컴포지션으로 하는가)
        • IS - A
          • A laptop IS - A computer / ~은 ~의 한 종류이다
          • Inheritance (상속)
          • 조건 1 : 모든 멤버와 모든 메서드의 중복이 되어야함
          • 재사용성(reusability) -> 재사용성 만으로 IS -A 상속을 하면 절대 안되고 조건을 모두 충족해야함!
        • HAS - A
          • A policeman has a gun / ~이 ~을 가지고 있다. or 포함하고 있다
          • Composition(합성) -> ◆(In UML)
            • same life cycle = 강하게 연결되어 있다
          • Aggregation(통합) -> ◇(In UML)
            • NOT same life cycle = weak coupling
    3. method overriding (메소드 오버라이딩) -> 상속이 필수
      • polymophism (다형성)
        • 상속 관계에 있는 다양한 클래스의 객체에서 같은 이름의 메서드를 호출할 때, 각 객체가 서로 다르게 구현된 메서드를 호출함으로써 서로 다른 행동, 기능, 결과를 가져오는 것
    4. abstract class (추상 클래스) = 자바에서의 인터페이스
      • 인스턴스를 만들 수 없다.
      • 하나 이상의 abstract method (추상 메소드)
        • 몸체(body)가 없는 함수 -> pass
      • 파생 클래스에서 반드시 overriding을 해야함 -> 구현을 강제하도록 한다

문자와 문자열, 함수전달 방식

  1. 문자, 문자열 -> character set, ASCII( -> UNI-CODE(테이블로 이루어져 있음)-> UTF-8, UTF-16, UTF-32)

    • character set -> 문자 집합(Ex. 라틴문자)

    • character encoding -> 문자 인코딩(Ex. 모스 부호)

    • code point -> 문자 하나에 정수 하나를 매핑해둠(Ex. 아스키 코드표(7bit로 모두 표현가능)

    • UniCode -> 2byte 기반으로 만듬(2byte 테이블을 17개 만들어놓았고 현재 실생활에서 쓰는건 2개(4개 등록되어있음))

      • 216개만큼의 문자가 매핑되어있음 BMP -> BMP까지는 2byte면 다 표현이 가능
      • 아스키는 코드포인트 자체 그대로 메모리에 저장함
      • 유니코드는 메모리에 저장할때 utf-8, utf-16, utf-32로 나눠서 저장
      • UTF-16의 경우 BMP에 해당하는 문자는 그대로 전송(2byte), 2차평면 이상부터는 4byte로보내는 대신 앞을 0으로 다 채워서 보냄
      • UTF-32는 처음부터 4byte로 바꿔서 보냄
      • UTF-8 -> 가변길이 인코딩, 한글은 3byte로 인식, 엔디언 변환을 무시(Endianness)하고 그대로 저장, 전송을 함

    • 사람들이 인식하는 것은 빅엔디언형식이고 컴퓨터는 대부분 Intel CPU를 사용하고 intel은 리틀엔디언 방식으로 저장함

U+AC00 을 메모리에 저장할때는 리틀엔디언 방식을 통해 00 AC로 저장함
네트워크를 통해 데이터를 보낼때는 빅엔디언으로 보냄
실제로 네트워크를 통해 전달되는 방식으로 변환되는것이 유니코드인코딩방식?

  1. 함수를 호출할때 인자 전달 방식 ->> call by value, call by reference -> C언어 call by object reference(call by assignment) -> 파이썬
    • call by value : 값에 의한 전달 -> 함수 스택프레임 안에서 값이 바뀌어도 밖의 스택프레임에는 영향을 주지않는것을 보장함
    • call by reference : 주소에 의한 전달 -> 함수 스택프레임 안에서 주소를 통해 바깥 스택프레임에 있는 변수에 영향을주어 값을 변화시킴
